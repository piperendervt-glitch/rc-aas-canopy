## Phase 4の計測指標（2026年3月16日追加）

```
① |G| = G_AAS × G_RC × G_Probe
   閉ループゲイン
   安定条件：|G| < 1

② λ₂（テッセラクット代数的連結度）
   16RCのネットワークの連結性

③ ベイズパラメータ実測値
   矛盾判定の信頼度

④ Probeあり/なし比較実験
   修正信号の有無による正答率・RC警告数の差

⑤ リアプノフ指数の推定
   G1実験のズレ時系列データから計算する
   正の指数 → カオス的
   負の指数 → 安定（アトラクターに収束中）
   ゼロ付近 → 臨界状態（崩壊の直前）
   → 閾値0.3の5つ目の根拠になる可能性

⑥ 安全関数S(x)の境界値の実測
   Probeの介入が最も効果的なズレの値を特定
   → 閾値0.3が安全関数の境界と一致するか確認

⑦ Probe介入量と効果の関係
   「大きな介入 vs 小さな早期介入」の比較
   OGY法的な微小介入が有効かどうか確認

⑧ P(En)の実測値（カスケーディングエラー）
   AAS→RC→Probeの各ステップのエラー増幅率
   参照：albatrosary (2026) の批判への実験的回答
   P(En) = 1-(1-p)^n の理論値と実測値を比較する

⑨ Cov(Xi, Xj)の推定（同調バイアス）
   AASとRCのエラーパターンの相関を測定する
   異なるモデルを使うことで相関がどれだけ下がるか確認
   → 実験2（タスク別モデル分離）の定量化

⑩ Foolの独立性スコア
   AAS・RCが同調したときに
   Foolが独立して検知できた割合
   = 「Foolをループの外に置く」設計の定量的根拠
   参照：Cov(Xi, Xj) > 0 への対策として論文で正当化できる
```

## Grok再評価（2026年3月16日）を受けたPhase 4設計変更

```
変更1：G₂群演算の形式的定義（最優先）
  「群条件を検証した」と言うためには
  「RC監視の演算」を形式的に定義する必要がある
  Phase 4で群演算を定義する実験を設計する
  → これが完成して初めて「群に着想を得た」を超えられる

変更2：RSI範囲をconstitutionで固定する
  Grokの指摘：「承認済み範囲のクリープ現象が避けられない」
  対策：承認済み範囲をconstitution.mdに明示的に記載する
  人間が承認できる変更の種類を事前に列挙して固定する
  → 範囲外の変更は自動的にPull Requestになる

変更3：Probeに別モデルを使う
  Grokの指摘：「同じ基盤モデルではCov > 0が避けられない」
  対策：
    AAS → qwen2.5:3b
    RC → ルールベース（LLM不使用）
    Probe → llama3.1:8b または mistral:7b（AASと別系列）
    Fool → llama3.1:8b（既に分離済み）
  → 同調バイアスを構造的に低減する

変更4：Stackelbergゲームの均衡崩壊リスクへの対処
  Grokの指摘：「固定リーダーが勝てないシナリオがある」
  対処方針：
    Adversarial AASが「勝った」記録も保存する
    → 「勝てなかった記録」だけでなく
      「勝てた場合の新しいFM」として記録する
    → 安全性証明にはならないが「未解決課題の発見」になる
    → constitution.mdに新しいFMとして追記する
```
